# 产品路线图

本路线图用于说明《国风炼金卡牌》从当前 MVP 原型到可发布版本的阶段安排。具体排期以后续人力、素材生产速度、抖音小游戏审核节奏和后端接入情况为准。

## Phase 0：产品与工程基线

目标：完成项目定位、玩法边界、配置结构与工程目录。

已完成 / 进行中：

- 明确项目方向为“国风历史卡牌收集游戏”。
- 明确“炼金”只指卡牌合成、碎片转化、升星和熔合。
- 建立 MVP 范围、技术栈、数据模型和 Cocos 前端结构文档。
- 建立 Cocos Creator 3.8.8 客户端工程。
- 建立秦汉 / 楚汉样例卡牌与合成链。
- 建立常驻基础池、周期朝代池、限时高级池配置雏形。

## Phase 1：MVP 单机闭环

目标：在 Cocos 客户端内跑通第一条完整卡牌收集闭环。

核心交付：

- 首页：当前周期卡池、卡册进度、今日任务入口。
- 抽卡页：单抽、十连、概率展示、主推卡牌。
- 抽卡结果：新卡、重复卡、稀有度表现。
- 合成页：历史关系合成、碎片合成、重复卡升星。
- 卡册页：按朝代、类型、稀有度筛选。
- 卡牌详情：历史简介、知识点、获得来源、合成线索。
- 本地存档：库存、碎片、星级、图鉴状态。

验收标准：

- 用户 3 分钟内理解抽卡、重复卡、碎片、升星、合成、卡册。
- 用户能通过秦汉 / 楚汉样例线合成至少一张高阶事件卡。
- 重复卡可以进入升星或碎片体系。
- 周期卡池和收集奖励能在本地配置中调整。

## Phase 2：内容与美术验证

目标：建立可扩展的国风历史卡牌内容生产方式。

核心交付：

- 完成至少 1 个朝代主题池的可展示素材。
- 建立卡牌主图、卡背、稀有度边框、Banner、背景纹样规范。
- 建立 ComfyUI 出图 prompt 模板与人工审核流程。
- 将资源路径绑定到卡牌配置。
- 优化“古籍炼炉”合成视觉，但不把项目改成修仙炼丹方向。

验收标准：

- 卡牌美术风格统一，能区分朝代、类型和稀有度。
- 抽卡结果、卡牌详情、卡册缩略图都能使用同一套资源。
- 素材命名、压缩、导入流程可重复。

## Phase 3：服务端与账号体系

目标：把核心经济与用户资产从本地迁移到服务端。

核心交付：

- 抖音小游戏登录与用户身份绑定。
- 用户卡牌、碎片、星级、卡册进度落库。
- 抽卡服务端权威校验。
- 合成、升星、分解、奖励领取服务端校验。
- 活动卡池配置读取与版本管理。
- 基础日志、异常检测和操作审计。

验收标准：

- 刷新或换设备后用户资产可恢复。
- 抽卡结果可追踪、可回放、可审计。
- 卡池概率和奖励不依赖客户端硬编码。

## Phase 4：商业化与运营系统

目标：接入抖音小游戏运营能力，形成可持续留存与变现。

核心交付：

- 每日任务、周任务、周期收集奖励。
- 激励视频广告奖励。
- 支付订单、抽卡券、礼包与首充策略。
- 运营后台：卡牌、卡池、奖励、活动配置。
- 数据看板：留存、抽卡、转化、卡册进度、广告收益。

验收标准：

- 可以无发版调整卡池、奖励和活动。
- 广告奖励与支付奖励发放可校验。
- 基础运营数据可用于下一轮内容调整。

## Phase 5：平台发布与增长

目标：完成抖音小游戏发布准备与首轮外部测试。

核心交付：

- 抖音小游戏包体与分包优化。
- 资源 CDN 或对象存储接入。
- 平台审核材料。
- 新手引导、首日留存优化。
- 首轮灰度测试和反馈收集。

验收标准：

- 首包体积、加载速度和兼容性达到平台要求。
- 新用户能顺利完成首次抽卡、合成和卡册查看。
- 有明确的数据指标支撑下一轮迭代。

## 暂不进入路线图的范围

- 复杂 PVP。
- 交易市场。
- 公会系统。
- 大地图探索。
- 装备养成。
- 长篇剧情战斗。
- 重度修仙 / 洞府 / 炼丹炉系统。
