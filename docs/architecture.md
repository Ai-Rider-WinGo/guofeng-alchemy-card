# 系统架构

《国风炼金卡牌》的工程架构围绕“配置驱动的卡牌收集游戏”展开。MVP 阶段优先用本地 JSON 和 Cocos 客户端完成主闭环，后续再接入服务端、运营后台、素材 CDN 与数据分析系统。

## 架构目标

- 让玩法规则从 UI 中解耦，便于调整卡池、概率、奖励、合成与活动节奏。
- 让卡牌内容、历史关系、图鉴进度与素材路径形成统一数据源。
- 让抖音小游戏客户端先完成轻量闭环，后续可以逐步迁移到服务端权威校验。
- 让 AI 出图、人工审核、资源压缩、Cocos 导入形成可追踪素材管线。

## 分层设计

```text
└─────────────────────┬──────────────────────┘
                      │
┌─────────────────────▼──────────────────────┐
│ 客户端玩法层                                  │
│ DrawManager / MergeManager / Inventory 等    │
└─────────────────────┬──────────────────────┘
                      │
┌─────────────────────▼──────────────────────┐

                      │

```

## 客户端职责

MVP 阶段，Cocos 客户端负责：

- 加载本地 JSON 配置。
- 展示首页、抽卡、合成、图鉴等核心页面。
- 执行本地抽卡、合成预览、库存变化和图鉴进度计算。
- 为抽卡动画、卡牌翻转、稀有度表现和国风 UI 做视觉承载。
- 为后续服务端接入保留接口边界。

当前脚本骨架：

- `ConfigLoader`：统一加载 `resources/config` 配置。
- `CardManager`：卡牌目录缓存与查询。
- `InventoryManager`：库存增加、移除、数量查询。
- `DrawManager`：按卡池与稀有度权重抽卡。
- `MergeManager`：按历史关系合成事件卡。
- `StarManager`：同名同星重复卡升星。
- `CollectionManager`：卡册解锁与收集进度。

## 服务端职责

生产版本中，服务端需要逐步接管以下权威逻辑：

- 用户登录与抖音平台身份绑定。
- 抽卡概率、抽卡消耗、抽卡结果落库。
- 用户背包、碎片、升星材料、卡册进度。
- 合成、分解、升星与奖励领取校验。
- 广告激励奖励发放与支付订单校验。
- 活动卡池、限时 UP、周轮换与补偿发放。
- 行为日志、异常检测、风控与数据看板。

## 配置流转

```text
策划案 / Obsidian
  -> GitHub docs
  -> config/*.json
  -> client/cocos-client/assets/resources/config/*.json
  -> Cocos ConfigLoader
  -> 客户端玩法 managers
```

短期可以手动同步配置；中期应增加同步脚本；生产阶段应由后台管理系统生成版本化配置并下发到客户端或服务端。

## 素材流转

```text
卡牌设定
  -> prompt 草稿
  -> ComfyUI 出图
  -> 人工筛选
  -> 命名与压缩
  -> Cocos resources / 远端对象存储
  -> card image_path 绑定
```

素材命名建议：

```text
{dynasty}_{type}_{name}_{rarity}_{version}.png
```

示例：

```text
qinhan_person_liubang_rare_v001.png
qinhan_event_hongmenyan_epic_v001.png
```

## 版本演进

- **MVP Pre-Alpha**：本地 JSON + Cocos 客户端闭环。
- **Alpha**：接入基础后端，抽卡与库存由服务端校验。
- **Beta**：接入运营后台、活动配置、广告激励与数据埋点。
- **Soft Launch**：打磨经济系统、留存节奏、素材质量与平台发布流程。
